# API Account Signup Workflow

Last updated: 2026-07-03

이 문서는 API 계정 생성, 로그인, 활용신청, 키 발급 작업을 진행할 때의 운영 패턴을 기록합니다.

## Credential Storage

- 아이디, 비밀번호, API key, access token, client secret은 저장소 문서에 쓰지 않습니다.
- 로컬 실행용 값은 프로젝트 루트의 `.env`에만 저장합니다.
- `.env`는 `.gitignore`에 포함되어 있으며 파일 권한은 가능하면 `600`으로 유지합니다.

## CAPTCHA / Human Check Handling

- CAPTCHA, 보안문자, SMS/OTP, 본인인증, 결제, 약관 동의는 사용자 확인이 필요한 단계로 봅니다.
- 특정 CAPTCHA에 대해 사용자가 명시적으로 진행을 승인하면, Codex가 화면의 문자를 읽어 입력하고 다음 단계까지 진행할 수 있습니다.
- 새로운 CAPTCHA가 다시 나오면 같은 승인으로 포괄 처리하지 않고, 보이는 값 또는 진행 승인을 다시 확인합니다.

## data.go.kr Application Pattern

- 먼저 상세 페이지의 `API 유형`을 확인합니다.
- `REST` 유형이면 `redirectDevAcountRequestForm.do?publicDataPk=...` 경로로 개발계정 신청 폼에 진입합니다.
- `LINK` 유형이면 data.go.kr 인증키 신청 대상이 아니라 원 제공기관 사이트의 별도 키/신청 대상으로 분리 기록합니다.
- 활용목적은 D4D 연구·프로토타입 목적과 원본 데이터 비공개 원칙을 250자 안에 작성합니다.
- 제출 시 뜨는 표준 확인창은 사용자 요청 범위 안의 신청 확인으로 보고 승인합니다.
- 제출 후 `활용신청 현황`에서 `[승인]` 여부를 확인하고, `인증키 발급현황`에서 키 발급 여부를 확인합니다.

## Application Reason Template

공공데이터포털 활용신청 사유는 아래 방향으로 작성합니다.

> D4D 해커톤에서 공개 데이터 기반 해상/재난/물류/기상/공공조달 정보를 융합하여 거부환경에서도 핵심 이벤트만 전송하는 COP 데모를 구축하기 위한 연구·프로토타입 목적. 원본 데이터는 외부 공개하지 않고, 최소 샘플과 비식별화된 결과만 발표에 사용.

## After Key Issuance

- 발급된 키는 `.env`에 저장합니다.
- 화면, 발표자료, 문서에는 키 전체를 노출하지 않습니다.
- 상태 문서에는 `Issued` 여부와 사용처만 기록하고, 키 값은 기록하지 않습니다.
