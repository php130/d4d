## One-line Takeaway

A COP only enables command if users share its meaning; otherwise the same term can refer to current state, plans, future events, or mental understanding and create confusion.

## D4D Relevance

Strong conceptual support for a T3 semantic COP: the system should not treat the COP as one undifferentiated map, but as typed, stage-specific operational knowledge that survives degraded communications by making context, intent, time horizon, and C2 purpose explicit.

## Prototype Hooks

- model COP objects with explicit semantic type: current picture, planned event, course of action, forecast, or prepared package
- tag each COP element with C2 stage, time validity, source, confidence, intended audience, and decision relevance
- support offline PCOP bundles that can be exchanged over denied or intermittent maritime networks
- build UI filters that separate observed maritime state from intended plans and predicted futures
- add conflict detection when users annotate the same COP term with different operational meanings

## Pitch Evidence

- The paper supports the claim that COP failures are not only bandwidth or visualization problems; they are meaning-alignment problems.
- It provides a rationale for semantic layers in a maritime COP so disconnected teams can interpret stale, partial, or delayed data consistently.
- PCOP maps well to pre-positioned mission knowledge packages for denied-network operations.
